# Pg

## Overview
`Pg` is an engine namespace: implemented natively by the game engine, not by any decompiled `.lua` module. It lives under no source file in `resident/` — it requires no `import()` and is always available as a global table from any script. It is a broad grab-bag covering object spawning, proximity-based object collection queries (`FastCollect*`), GUID-by-name lookup, the pursuit/wanted-level system, mission contract lifecycle, achievements, static-layer/asset streaming, and miscellaneous world queries (boundary radius, tether points, landing zones, camera-relative points).

## Provenance
The 80 functions listed below are a complete, authoritative enumeration taken from a live `pairs(Pg)` dump in-game — every name here is confirmed to exist. That dump gives names and raw function pointers only, nothing about parameters or behavior. Everything beyond that (argument shapes, likely purpose) is inferred from real call sites in the ~230 decompiled `.lua` files; where no call site exists anywhere in that corpus, this page says so explicitly rather than guessing.

`Pg.Spawn` and `Pg.FastCollectGroundVehicles` are already documented elsewhere in this wiki with real call-site evidence; see the cross-links in the relevant tables below rather than re-deriving them here.

### Pursuit/Wanted System

Live-confirmed 2026-07-22 via WebSocket lua-bridge probe against a running game — most of this table moves from bare name-only enumeration to tested behavior. Factions resolve to a guid the normal way, via `Pg.GetGuidByName("Allied"|"VZ"|"OC")`. Two rows below carry real gotchas expanded on in **Notes for modders**: `SetMaxPursuitLevel` (one-way ratchet) and `ClearPursuitLock` (the actual reset).

| Function | Signature (best-known) | Notes |
|---|---|---|
| `GetPursuitState` | `GetPursuitState()` | Used with no arguments: `local tCurrentPursuit = Pg.GetPursuitState()` (`resident/mrxfactionmanager.lua`). **Live-confirmed** via WebSocket lua-bridge probe (2026-07-22): this is the read channel for the whole system — returns a table shaped `{Level, Active, PlayerState, Faction, Locked, SecondsLeft, SecondsLeftInLevel, Duration}`. ## Notes for modders

**Unresolved spawn-by-name calls hard-crash the game — there is no soft failure.** This engine inlines its argument checks, so missing or wrong-typed arguments are safe: almost everything just returns `nil` silently (no Lua error) when called wrong, which is why it's generally safe to probe an unfamiliar function blind. Name resolution failures are a different, much more dangerous case: calling `Pg.Spawn` — or other name-based setters elsewhere in the engine, like `SetOutfit`/`SetModelName` — with a template/outfit/model name string the engine can't resolve is confirmed to hard-crash to desktop, live-confirmed 2026-07-22 on `Pg.Spawn('Austin (CIV)', ...)`. There's no exception to catch and nothing to recover in-script once it happens. Always gate a name-based spawn behind `Pg.GetGuidByName(sName) ~= nil` first rather than trying a name and seeing what happens.

**`SetMaxPursuitLevel` is a one-way ratchet DOWN for the rest of the session.** Live-confirmed 2026-07-22: `Pg.SetMaxPursuitLevel(n)` caps the pursuit level ceiling — a `Pg.SetPursuit(uFaction, 5)` called right after `SetMaxPursuitLevel(2)` only reached `Level=2` — but nothing raises that ceiling back up in the same session. Every argument form tried failed to undo it: a bigger `n` to `SetMaxPursuitLevel` itself, a `(uFaction, n)` form, `99`, `SetPursuitLevelTimes`, and `SetCustomPursuit` were all tried and none worked. The only way to reset the ceiling is a save-load or a full game restart — treat any call to this function as a one-way trapdoor for the current session, not a normal tunable setting.

**`ClearPursuitLock` is the real reset for an active pursuit; the `Restrict*`/`ClearPursuitRestrictions` functions are not.** Live-confirmed 2026-07-22: `Pg.ClearPursuitLock(true)` drops an active pursuit back to `Level=0`, inactive — that's the actual off switch. By contrast, `RestrictAllPursuit(false)` and `ClearPursuitRestrictions()` are both confirmed to not clear an already-active pursuit despite the names sounding like they would. `RestrictAllPursuit(b)` and `RestrictPursuitFaction(uFaction, b)` also don't block a scripted `Pg.SetPursuit(...)` call at all — both execute cleanly but only gate organic (AI-driven) pursuit escalation, not a direct `SetPursuit` override. If a pursuit needs to actually stop, call `ClearPursuitLock(true)`; the restriction functions only prevent new organic heat from building.

The `FastCollect*` family — `FastCollectHumans`, `FastCollectGroundVehicles`, `FastCollectBuildings`, `FastCollectFlying`, `FastCollectTanks`, `FastCollectHelicopters`, plus `Boats`/`Cars`/`Jets`/`Props`/`Usables`/`GroundVehiclesExceptTanks` (all six live-confirmed via WebSocket lua-bridge probe 2026-07-22, see table above, though still with zero real corpus call sites) — is the main building block used throughout the decompiled mission scripts for "find nearby objects of type X" logic — e.g. finding all soldiers of a faction near a point, all occupied buildings near a target, or all ground vehicles near a spawn point before filtering targets. The shared shape is `(x, y, z, fRadius, sFilter?)`, with the filter argument confirmed optional (multiple functions in the family have call sites both with and without it) and confirmed to sometimes be a non-string object rather than a plain string (see `FastCollectGroundVehicles` in `resident/mrxharmstrike.lua`, which passes `oFilter`). Anything building a "nearby objects" helper should model itself on this pattern; see also `resident/alarm.lua`, `resident/outpost.lua`, and `resident/mrxutil.lua` for real usage in context.

`GetGuidByName` — flagging the aliasing question directly: a live `pairs(_G)` scan (per prior work in this project) found `GetGuidByName` and `GetAllGuidsByName` present as bare loose globals on `_G`, in addition to being members of `Pg`. Grepping the decompiled corpus confirms both call forms exist for `GetGuidByName`: the namespaced `Pg.GetGuidByName(...)` is used in the large majority of call sites, but a substantial number of scripts (`vz/allcon001.lua`, `vz/chicon002.lua`, `vz/gurcon001.lua`, `vz/gurcon002.lua`, `vz/oilcon002.lua`, `vz/oilcon021.lua`, `vz/pmccon003.lua`, and others) call the bare, unqualified `GetGuidByName(...)` directly. The strongest evidence is in `vz/allcon001.lua` lines 77-81, where the same lookup (`GetGuidByName("China")`) and `Pg.GetGuidByName(...)` calls appear interleaved in adjacent statements within the same function, operating on equivalent name strings — strongly suggesting these are the same underlying function reachable at two scopes (i.e. `GetGuidByName` is aliased onto `_G` in addition to living on `Pg`), rather than two distinct implementations. This is inference from consistent call-site behavior, not confirmed by reading engine source (which isn't available) — but it is the most plausible reading of the evidence. `GetAllGuidsByName` by contrast has zero call sites under either form anywhere in the decompiled corpus — its signature and return shape come from a direct 2026-07-22 WebSocket lua-bridge probe against a running game instead (see the table above), not from corpus evidence.
